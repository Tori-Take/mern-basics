マルチテナント・アーキテクチャ実装計画

TODOアプリをスケーラブルなSaaSへと進化させるためのロードマップ

はじめに：ゴール設定

現在のTODOアプリを、複数の組織（テナント）が安全に共存できるマルチテナント・アプリケーションへと進化させます。最終的には、組織の親子関係を反映した階層型マルチテナントを実現し、「上位組織が下位組織のデータを閲覧できる」といった高度なアクセス制御を実装することを目指します。

フェーズ1基本的なマルチテナント基盤の構築

まず、各テナントが完全に独立して動作する、最も基本的なマルチテナントの仕組みを実装します。これが全ての土台となります。

ステップ1: Backend `Tenant`モデルの作成

アプリケーションを利用する組織（会社、部署、家族など）の単位となる「テナント」の設計図を作成します。

  • server/models/tenant.model.jsを新規作成します。
  • テナント名（name: String）を持つシンプルなスキーマを定義します。

ステップ2: Backend 既存モデルへの`tenantId`追加

全てのデータが「どのテナントに所属するか」を明確にするため、各モデルに所有者情報を刻印します。

  • user.model.js, todo.model.js, role.model.jsを修正します。
  • 各スキーマにtenantId: { type: Schema.Types.ObjectId, ref: 'Tenant', required: true }を追加します。

ステップ3: Backend/Frontend ユーザー登録フローの刷新

ユーザーが登録する際に、自身が所属するテナントも同時に作成・決定するフローを構築します。

  • バックエンド: 新規登録APIを改修し、ユーザー情報と共に「組織名（テナント名）」を受け取れるようにします。内部で新しいテナントを作成し、そのIDを新しいユーザーに紐付けます。
  • フロントエンド: 登録ページ(RegisterPage.jsx)に「組織名」の入力フィールドを追加します。

💡 最重要：テナント分離ロジックの実装

ここがマルチテナントの心臓部です。APIリクエストのたびに、「ログインユーザーのテナントに属するデータしか操作させない」という絶対的なルールを強制します。

ステップ4: Backend APIの全面改修

全てのデータベース操作に、テナントによる絞り込みを適用します。

  • データ取得 (GET): Todo.find() を Todo.find({ tenantId: req.user.tenantId }) のように変更し、自テナントのデータのみを返します。
  • データ作成 (POST): 新しいデータを作成する際、new Todo({ ...data, tenantId: req.user.tenantId }) のように、自動で自テナントIDを付与します。
  • データ更新/削除 (PUT/DELETE): 操作対象のデータを検索する際、Todo.findOne({ _id: req.params.id, tenantId: req.user.tenantId }) のように、IDとテナントIDの両方で絞り込み、他テナントのデータを誤って操作することを防ぎます。

フェーズ2階層型マルチテナントへの進化

基本的なテナント分離が完成したら、テナント間に親子関係を持たせ、より複雑な組織構造に対応できるようにします。

ステップ5: Backend `Tenant`モデルの階層化

テナントが「親」を持てるように、モデル自身を参照するフィールドを追加します。

  • tenant.model.jsを修正します。
  • parent: { type: Schema.Types.ObjectId, ref: 'Tenant', default: null }フィールドを追加します。

ステップ6: Backend アクセス制御ロジックの高度化

「自分自身」と「全ての子孫」のテナントにアクセスできるように、権限チェックロジックを拡張します。

  • バックエンドに、指定されたテナントID配下にある全ての子孫テナントIDを再帰的に検索し、リストを返すヘルパー関数 (例: getAccessibleTenantIds) を作成します。
  • データ取得API (GET) を改修し、find({ tenantId: { $in: accessibleTenantIds } }) のように、Mongooseの$in演算子を使って、アクセス可能な全てのテナントのデータを一度に取得できるようにします。

フェーズ3管理機能とUI/UXの洗練

システムが完成したら、管理者がテナントやユーザーを管理しやすくするためのUIを構築します。

ステップ7: Frontend テナント管理UIの作成

管理者がテナントの階層構造を視覚的に把握し、編集できるようにします。

  • 管理者ダッシュボードに「テナント管理ページ」を新設します。
  • テナントのツリー構造を表示し、ドラッグ＆ドロップで親子関係を変更できるようなUIを検討します（ライブラリ利用も視野に）。
  • 新しい子テナント（部署など）を作成する機能を追加します。

ステップ8: Backend/Frontend ユーザー招待機能の実装

既存のテナントに新しいユーザーを招待する、より実践的なフローを構築します。

  • 管理者がメールアドレスを入力して、自分のテナントにユーザーを招待できるUIを作成します。
  • バックエンドで招待トークンを生成し、招待メールを送信する仕組みを実装します（nodemailerなど）。
  • 招待されたユーザーは、メール内のリンクからパスワードを設定して登録を完了します。
