MERNスタック学習記録 - コントローラー層の導入編

はじめに

バックエンドのDDD（ドメイン駆動設計）への再構築が完了し、次のステップとして、さらなるコード品質の向上を目指しました。
今回のリファクタリングでは、「コントローラー層」を導入し、ルーティングとビジネスロジックの責務を明確に分離することに成功しました。

課題：ルートファイルの肥大化

リファクタリング以前は、users.routes.jsのようなルートファイルが、リクエストの受付（ルーティング）だけでなく、データベースへのアクセス、データの加工、エラーハンドリングといったビジネスロジックのすべてを担っていました。

これにより、ファイルは肥大化し、コードの見通しが悪くなっていました。どの部分が何を担当しているのかを把握するのが難しく、将来的な機能追加やメンテナンスが困難になるリスクを抱えていました。

解決策：コントローラー層の導入

この問題を解決するため、identityドメインにusers.controllers.jsファイルを新設しました。

  • ルート (users.routes.js): 「どのURLパスに、どのHTTPメソッドでアクセスされたか」という交通整理のみを担当します。具体的な処理は行わず、対応するコントローラーのメソッドを呼び出すだけです。
  • コントローラー (users.controllers.js): ルートから渡されたリクエスト(req)とレスポンス(res)を使い、実際のビジネスロジック（ユーザー検索、パスワード検証、DB保存など）を実行します。

💡 Before (ルートファイル)

// ログイン処理の例
router.post('/login', async (req, res) => {
  // ユーザー検索、パスワード比較、JWT生成などのロジックがここに記述されていた
});

✨ After (ルートファイル)

// ロジックはUserControllerに委譲
router.post('/login', UserController.login);

達成された成果と次のステップ

コントローラー層の導入により、以下の大きな成果を得ることができました。

  • 責務の分離: ルーティングとビジネスロジックが明確に分かれ、コードの可読性と保守性が劇的に向上しました。
  • テストの容易性: ビジネスロジックが純粋な関数（メソッド）として分離されたことで、単体テストが書きやすくなりました。
  • 再利用性の向上: 複数のルートから同じロジックを呼び出すといった再利用が容易になりました。

このリファクタリングは、堅牢でスケーラブルなバックエンドアーキテクチャへの重要な一歩です。
次のステップでは、計画書refactoring-next-steps.htmlに従い、「リポジトリパターンの導入」に進みます。これにより、ビジネスロジックとデータアクセスロジックをさらに分離し、よりクリーンなコードベースを目指します。
