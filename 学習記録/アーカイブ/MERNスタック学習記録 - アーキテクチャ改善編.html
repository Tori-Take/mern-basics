MERNスタック学習記録 - アーキテクチャ改善編

はじめに

DDD（ドメイン駆動設計）に基づいたファイル構成の再構築に続き、バックエンドの内部品質を劇的に向上させるための大規模なリファクタリングを実施しました。
この取り組みにより、「コントローラー層」と「リポジトリパターン」という、現代的なソフトウェアアーキテクチャに不可欠な2つの設計パターンを導入することに成功しました。

ステップ1：コントローラー層の導入

課題： *.routes.js ファイルが、APIのルーティングとビジネスロジックの両方を担当しており、コードが肥大化・複雑化していました。

解決策： ビジネスロジックを *.controllers.js ファイルに分離しました。これにより、各ファイルの責務が明確になりました。

  • ルート (*.routes.js): HTTPリクエストの交通整理に専念。
  • コントローラー (*.controllers.js): 実際のビジネス処理を実行。

💡 Before (ルートファイル)

router.post('/login', async (req, res) => {
  // ユーザー検索、パスワード比較、JWT生成などのロジックがここに混在
});

✨ After (ルートファイル)

router.post('/login', UserController.login);
⬇

ステップ2：リポジトリパターンの導入

課題： コントローラー層を導入した後も、ビジネスロジック内に User.findOne() や user.save() といったデータベース操作のコードが直接記述されており、ビジネスロジックが特定のデータベース技術（Mongoose）に依存していました。

解決策： データベースとの対話をすべてカプセル化する *.repository.js ファイルを作成しました。コントローラーは、リポジトリのメソッドを呼び出すだけで、データの永続化方法を意識する必要がなくなりました。

💡 Before (コントローラー)

// ビジネスロジック内にDB操作が直接書かれている
const user = await User.findById(req.user.id);

✨ After (コントローラー)

// DB操作はリポジトリに委譲
const user = await userRepository.findById(req.user.id);

達成された成果：クリーンアーキテクチャの実現

この2段階のリファクタリングにより、バックエンドは責務が明確に分離された3層アーキテクチャへと進化しました。

  1. ルーティング層 (*.routes.js): HTTPリクエストの受付。
  2. コントローラー層 (*.controllers.js): ビジネスロジックの実行。
  3. リポジトリ層 (*.repository.js): データベースとの通信。

この構造は、コードの可読性、保守性、そして何よりもテストの容易性を劇的に向上させます。将来的にデータベースを変更する場合でも、ビジネスロジックに影響を与えることなく、リポジトリ層の修正だけで対応可能になりました。これは、プロフェッショナルな開発現場で採用されている、非常に堅牢でスケーラブルな設計です。
