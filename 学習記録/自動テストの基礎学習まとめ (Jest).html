自動テストの基礎学習まとめ (Jest)

1. 自動テストの目的とメリット

「テストコード」という仕様書を書いておき、本体のコードを変更した際に、その仕様通りに動くかを自動で確認する仕組み。

  • リファクタリングへの安心感：コードを綺麗にしても、既存の機能を壊していないことを保証できる。
  • デグレード（機能退行）の防止：新しい機能追加によって、古い機能が壊れるのをすぐに検知できる。
  • 生きたドキュメント：テストコードが、コードの正しい使い方や仕様を示す「動く仕様書」になる。

2. Jestの基本構造

Jestは、テストを記述するための便利な関数をグローバルに提供します。

describe(name, fn)

関連するテストをグループ化します。このグループを「テストスイート」と呼びます。

it(name, fn) または test(name, fn)

個々のテストケースを定義します。「このテストは何を検証するのか」をnameに記述します。

expect(value).matcher(expectedValue)

テストの核となる部分。「valueがexpectedValueという期待値と一致すること」を表明（アサーション）します。

  • .toBe(value): 厳密な等価性 (===) をチェックします。プリミティブな値（数値、文字列など）の比較に使います。

3. ソースコードとテストコードの分離

自動テストの基本は、「テストされる側のコード」と「テストする側のコード」を別のファイルに分けることです。

テスト対象のファイル (utils.js)

テストしたい実際の機能（関数など）を定義し、module.exportsで外部に公開します。

// utils.js
const sum = (a, b) => {
  return a + b;
};

module.exports = {
  sum,
};

テストファイル (utils.test.js)

テスト対象の機能をrequireでインポートし、その振る舞いを検証します。

// utils.test.js
const { sum } = require('./utils.js');

describe('sum function', () => {
  it('should return the sum of two positive numbers', () => {
    expect(sum(1, 2)).toBe(3);
  });
});

4. テストの実行と命名規則

テストの実行コマンド

package.jsonのscriptsに定義されたコマンドを実行します。

npm test

これは、package.json内の"test": "jest"という定義に基づいてjestコマンドを実行しています。

テストファイルの命名規則

Jestは以下のパターンのファイルを自動でテスト対象として認識します。

  • ファイル名が .test.js で終わる (例: utils.test.js)
  • ファイル名が .spec.js で終わる (例: utils.spec.js)
  • __tests__/ ディレクトリの中にあるJSファイル
